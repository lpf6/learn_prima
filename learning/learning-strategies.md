## 📖 如何高效阅读代码

### 方法 1：自顶向下法

**适用场景**：理解整体架构

**步骤**：
1. **从 main 函数开始**
   ```bash
   # 找到入口
   grep -r "int main" examples/main/
   ```

2. **追踪调用链**
   ```
   main → llama_init → llama_decode → llama_output
   ```

3. **画出流程图**
   ```mermaid
   graph TD
   A[main] --> B[初始化模型]
   B --> C[处理 prompt]
   C --> D[生成 token]
   D --> E[输出结果]
   ```

4. **理解数据流**
   - 输入是什么？
   - 经过哪些处理？
   - 输出是什么？

---

### 方法 2：自底向上法

**适用场景**：理解核心算法

**步骤**：
1. **找到核心 kernel**
   ```bash
   # 找到关键文件
   find . -name "*.cu" | grep -E "(mmq|fattn|norm)"
   ```

2. **逐行分析**
   ```cpp
   // 示例：分析矩阵乘法
   // 1. 线程索引计算
   int row = blockIdx.y * blockDim.y + threadIdx.y;
   int col = blockIdx.x * blockDim.x + threadIdx.x;
   
   // 2. 加载数据到共享内存
   As[ty][tx] = A[row * K + tx];
   
   // 3. 同步
   __syncthreads();
   
   // 4. 计算
   acc = A * B + C;
   ```

3. **做笔记**
   - 每行代码的作用
   - 为什么这样写
   - 有没有更好的方式

4. **画图理解**
   - 内存布局图
   - 线程执行图
   - 数据流图

---

### 方法 3：对比阅读法

**适用场景**：学习不同实现方式

**步骤**：
1. **找到对比代码**
   - CUDA 实现 vs TileLang 实现
   - CPU 实现 vs GPU 实现
   - 优化前 vs 优化后

2. **对比维度**
   | 维度 | CUDA | TileLang |
   |------|------|----------|
   | 代码行数 | 200+ | 30 |
   | 抽象层次 | 低 | 高 |
   | 性能 | 100% | 95% |
   | 可读性 | 低 | 高 |

3. **总结优劣**
   - 各自的优势
   - 各自的局限
   - 适用场景

## 🎯 如何设定目标

### SMART 原则

**S - Specific（具体的）**
- ❌ 坏目标："学习 CUDA"
- ✅ 好目标："理解 CUDA 线程模型和内存层次"

**M - Measurable（可衡量的）**
- ❌ 坏目标："提高性能"
- ✅ 好目标："将矩阵乘法性能提升 50%"

**A - Achievable（可实现的）**
- ❌ 坏目标："一周学完所有内容"
- ✅ 好目标："两周完成第一阶段学习"

**R - Relevant（相关的）**
- ❌ 坏目标："学习图形渲染"（与目标无关）
- ✅ 好目标："学习矩阵乘法优化"（直接相关）

**T - Time-bound（有时限的）**
- ❌ 坏目标：" someday 完成项目"
- ✅ 好目标："3 个月内完成推理引擎项目"

## 🔍 如何发现问题

### 5 Why 分析法

**问题**：程序运行很慢

连续问 5 次"为什么"：

1. **为什么慢？** → GPU 利用率低
2. **为什么利用率低？** → 线程数太少
3. **为什么线程数少？** → Block 大小设置不合理
4. **为什么设置不合理？** → 没有根据硬件调整
5. **为什么没调整？** → 不了解硬件特性

**根本原因**：缺乏硬件架构知识

**解决方案**：学习 GPU 架构章节

## 🛠️ 如何解决问题

### 橡皮鸭调试法

**步骤**：
1. 准备一个橡皮鸭（或任何物体）
2. 向鸭子解释你的代码
3. 一行一行地解释
4. 在解释过程中，往往会自己发现问题

**为什么有效？**
- 强迫你慢下来
- 换个角度思考
- 激活不同脑区

**示例**：
```
"好的，鸭子先生，让我解释这段代码...
这一行计算线程索引...
等等，这里好像有问题！
threadIdx.x 应该是 threadIdx.y..."
```

---

### 费曼技巧

**四步法**：

1. **选择概念**
   - 例如：Warp Shuffle

2. **教给别人**
   - 假装教一个完全不懂的人
   - 用最简单的语言
   - 避免专业术语

3. **发现漏洞**
   - 哪里卡住了？
   - 哪里解释不清楚？
   - 这些就是你的知识盲点

4. **重新学习**
   - 查阅资料
   - 填补盲点
   - 简化解释

**效果**：
- 深度理解概念
- 长期记忆
- 能够应用

## 🎭 思维模式

### 成长型思维 vs 固定型思维

| 固定型思维 | 成长型思维 |
|-----------|-----------|
| "我没有天赋" | "我可以通过努力提升" |
| "这太难了" | "这需要更多时间和方法" |
| "我失败了" | "我学到了什么" |
| "他好聪明" | "他一定很努力" |
| 避免挑战 | 拥抱挑战 |
| 轻易放弃 | 坚持不懈 |

**培养成长型思维**：
1. 把"我不会"改成"我还不会"
2. 把"我失败了"改成"我发现了一个行不通的方法"
3. 关注过程，而不是结果
4. 庆祝小进步

## 🆘 应对困难

### 5 分钟规则

**当你想放弃时**：
1. 告诉自己"再坚持 5 分钟"
2. 5 分钟后，如果还想放弃，就休息
3. 通常 5 分钟后，你已经进入状态

**原理**：
- 开始是最难的
- 一旦开始，就容易继续

---

### 分解问题

**大问题**：
```
"实现 Flash Attention" ← 太难了！
```

**分解成小问题**：
```
1. 理解 Attention 公式
2. 实现基础版本
3. 理解在线 Softmax
4. 实现分块计算
5. 优化内存访问
6. 性能测试
```

每个小问题都可解决！

---

### 寻求帮助

**何时求助**：
- 卡住超过 30 分钟
- 尝试了所有方法
- 情绪开始沮丧

**如何求助**：
1. 说明你做了什么
2. 说明你的理解
3. 提供错误信息
4. 提供复现步骤

**好的提问**：
```
"我在实现矩阵乘法，使用了共享内存优化，
但性能只有预期的 50%。

我做了：
1. 使用 16x16 的 tile
2. 避免了 bank conflict
3. 使用了 __syncthreads()

我用 Nsight 分析，发现 occupancy 只有 25%。

请问可能是什么原因？"
```

**坏的提问**：
```
"我的代码很慢，怎么办？"
```
